## ① SNSアプリ「Pulse」 — 感情共鳴型SNS（PHP / SQLite / JavaScript）

[![View Demo](https://img.shields.io/badge/▶_View_Live_Demo-blue)](http://shimada.atwebpages.com/pulse/)

Pulse は、一般的な SNS の「文章を投稿する」「他の人の投稿を見る」「反応する」という基本機能に、**感情を選んで投稿する**という要素を加えた Web アプリです。投稿時に「喜び」「愛」「穏やか」「活力」「悲しみ」「怒り」「驚き」「不安」から今の気分を選び、タイムライン上ではその感情が色や表示に反映されます。

初学者向けに言い換えると、Pulse は「会員登録できるミニSNS」です。ユーザーがフォームに入力した内容を PHP が受け取り、SQLite データベースに保存し、保存された投稿を一覧として画面に出しています。SNSらしい機能を作ることで、Web アプリ開発でよく使う処理を一通り学べる構成にしています。

### 利用者の流れ

1. 新規登録画面でユーザー名・メールアドレス・パスワードを入力します。
2. 登録後、ログインするとタイムラインが表示されます。
3. 投稿画面で本文と感情ムードを選び、投稿します。
4. タイムラインに自分や他のユーザーの投稿が表示されます。
5. 気になった投稿には「共鳴」リアクションを送り、プロフィールから投稿履歴や感情傾向を確認できます。

### 裏側の処理

ユーザー登録時には、パスワードをそのまま保存せず `password_hash()` でハッシュ化して保存します。ログイン時には `password_verify()` で入力されたパスワードと保存済みハッシュを照合します。投稿やリアクションは SQLite のテーブルに保存し、画面表示時に必要なデータを取り出して HTML として出力します。

「共鳴」やフォローなど、画面の一部だけを更新したい操作では JavaScript を使っています。ページ全体を再読み込みせずに反応が返るため、利用者には軽く動いているように感じられます。

### コードを読む順番

| 順番 | ファイル | 初学者向けの見どころ |
|------|----------|----------------------|
| 1 | `config/database.php` | SQLite への接続、`users` / `posts` / `resonances` / `follows` テーブル作成、外部キー、インデックスの考え方 |
| 2 | `includes/functions.php` | エスケープ、CSRF、ログイン判定、ムード一覧、共鳴数、フォロー数など、複数画面で使う共通処理 |
| 3 | `register.php` / `login.php` | フォーム送信、入力チェック、パスワードハッシュ化、セッション保存の流れ |
| 4 | `index.php` | 投稿一覧を SQL で取得し、PHP の `foreach` でカード表示する流れ |
| 5 | `post.php` | 投稿フォームから送られた本文・ムード・公開条件を DB に保存する流れ |
| 6 | `api/resonate.php` / `api/reply.php` / `api/follow.php` | JavaScript から呼ばれる Ajax API。画面遷移なしで状態を更新する仕組み |
| 7 | `public/js/app.js` / `public/css/style.css` | ボタンクリック時の非同期通信、表示切替、ムードごとの見た目調整 |

### 処理の流れをコードで追う例

投稿機能を追う場合は、まず `post.php` のフォームを見ます。`method="post"` で送信された内容は同じ `post.php` 内で受け取り、`verifyCSRFToken()` で正しい送信かを確認します。その後、`$pdo->prepare()` で SQL を準備し、本文やムードなどを `execute()` で安全に渡して `posts` テーブルに保存します。

共鳴ボタンは、`index.php` のボタンに `data-post-id` を持たせ、`public/js/app.js` の関数から `api/resonate.php` に送信します。API 側では「すでに共鳴しているか」を調べ、あれば削除、なければ追加します。最後に JSON を返し、JavaScript が件数とボタン表示を更新します。

### 主な実装機能

| 機能 | 初学者向けの説明 |
|------|----------------|
| ユーザー認証 | 会員登録・ログイン・ログアウトを行い、ログイン中のユーザーをセッションで管理 |
| ムード投稿 | 投稿本文に加えて感情カテゴリを保存し、表示色やフィルターに利用 |
| 共鳴リアクション | 「いいね」に近い機能を独自名で実装し、Ajax で反応を更新 |
| タイムラインフィルター | 感情別・フォロー中のみなど、条件に合う投稿だけを表示 |
| タイムカプセル投稿 | 未来日時を指定し、その時刻以降に表示される投稿を作成 |
| 感情天気予報 | 直近24時間の投稿ムードを集計し、コミュニティ全体の傾向を表示 |
| 感情オーラ | プロフィールに直近7日間の感情傾向を可視化 |
| ささやきモード | 匿名投稿として表示できる機能 |
| フォロー機能 | ユーザー同士のつながりを保存し、プロフィールで確認 |

### セキュリティ実装

| 対策 | 実装内容 |
|------|--------|
| SQLインジェクション対策 | PDO + プリペアドステートメントで SQL に値を安全に渡す |
| XSS対策 | `htmlspecialchars()` で投稿内容をエスケープして表示 |
| CSRF対策 | フォームに `csrf_token` を埋め込み、送信元を検証 |
| パスワード保護 | bcrypt ハッシュ化（`password_hash()` / `password_verify()`） |
| セッション管理 | PHP セッションでログイン状態を保持 |
| タイミング攻撃対策 | `hash_equals()` でトークンを安全に比較 |

- PHP でフォーム入力を受け取り、画面に反映する流れ
- SQLite と PDO を使ったデータベース操作
- ログイン機能とセッション管理の基本

## ② 掲示板アプリ（PHP / MySQL）

[![View Demo](https://img.shields.io/badge/▶_View_Live_Demo-blue)](http://shimada.atwebpages.com/post/)

掲示板アプリは、ユーザーがログインして文章を投稿し、他の投稿に返信できる Web アプリです。SNS よりも機能を絞り、「登録」「ログイン」「投稿」「返信」「削除」という Web アプリの基本操作を分かりやすく確認できる構成にしています。

初学者向けに言うと、掲示板は Web アプリ開発の土台を学ぶ題材です。フォームから送信された文字を PHP で受け取り、MySQL に保存し、保存済みの投稿を一覧表示します。ユーザー情報と投稿情報を別々のテーブルに分けることで、データベース設計の基本も学べます。

### 利用者の流れ

1. 新規会員登録で名前・メールアドレス・パスワードを登録します。
2. ログインすると投稿一覧ページに移動します。
3. 入力フォームから新しい投稿を作成します。
4. 他の投稿に返信すると、親投稿と関連した投稿として保存されます。
5. 自分の投稿は削除できます。

### 裏側の処理

投稿作成時は、入力された本文とログイン中ユーザーの ID を `posts` テーブルに保存します。返信の場合は、どの投稿への返信かを示す `reply_post_id` も保存します。表示時には `members` テーブルと `posts` テーブルを関連付け、投稿者名や投稿本文を一緒に表示します。

ログイン処理では、入力されたメールアドレスとパスワードを確認し、正しければセッションにユーザー情報を保存します。セッションにより、ページを移動しても「誰がログインしているか」を保持できます。

### コードを読む順番

| 順番 | ファイル | 初学者向けの見どころ |
|------|----------|----------------------|
| 1 | `db.example.php` | DB 接続設定の見本。実運用では秘密情報を含む `db.php` を Git 管理しない考え方 |
| 2 | `login.php` / `check.php` | CSRF トークン、ログインフォーム、`password_verify()`、セッション保存の流れ |
| 3 | `input.php` / `regist.php` | 会員登録フォームから `members` テーブルに保存する流れ |
| 4 | `index.php` / `index2.php` | 投稿一覧を表示する画面。DB から取得したデータを HTML に埋め込む方法 |
| 5 | `write.php` / `write2.php` | 通常投稿と返信投稿を DB に追加する処理 |
| 6 | `reply.php` | 返信対象の投稿を表示し、親投稿 ID を引き継いで返信を作る仕組み |
| 7 | `delete.php` | 投稿削除処理。対象 ID を受け取り、SQL の `DELETE` を実行する流れ |

### 処理の流れをコードで追う例

ログイン処理は `login.php` から読み始めると分かりやすいです。`login.php` はフォームと CSRF トークンを用意し、送信後は `check.php` がメールアドレスとパスワードを確認します。DB からユーザーを探すときは `prepare()` と `execute()` を使い、入力値を SQL 文字列へ直接連結しないようにしています。

投稿処理は `index.php` の入力欄から `write.php` へ進みます。`write.php` では、ログイン中ユーザーの ID と投稿本文を受け取り、`posts` テーブルへ `INSERT` します。返信の場合は `reply.php` で元投稿を表示し、`write2.php` で `reply_post_id` を保存するため、「どの投稿への返信か」が DB 上で分かるようになります。

### 主な実装

| 機能 | 初学者向けの説明 |
|------|----------------|
| ユーザー登録 | 入力された会員情報を MySQL に保存 |
| ログイン / ログアウト | セッションを使ってログイン状態を管理 |
| 投稿作成 | フォーム入力を受け取り、投稿としてデータベースに保存 |
| 返信投稿 | 元の投稿 ID を保存し、投稿同士の関係を表現 |
| 投稿削除 | 対象投稿をデータベースから削除 |
| カード型 UI | 投稿を読みやすいカード形式で表示 |
| レスポンシブ対応 | PC・タブレット・スマートフォンで見やすいレイアウト |

### セキュリティと入力処理

- `htmlspecialchars()` で投稿内容を安全に表示し、XSS を防止
- PDO のプリペアドステートメントで SQLインジェクションを防止
- CSRF トークンで、意図しないフォーム送信を防止
- ログインが必要な画面ではセッションを確認し、未ログインならログイン画面へ誘導

### この作品で学べること

- PHP と MySQL を使った基本的な CRUD 処理
- 会員情報と投稿情報を別テーブルで管理する考え方
- ログイン状態をセッションで保持する仕組み
- 返信機能のような「データ同士の関連」の作り方
- 入力値を安全に扱うための基本的な対策

## ③ 定型文管理アプリ（Python / Flet）

定型文管理アプリは、よく使う文章を保存しておき、必要なときにワンクリックでコピーできるデスクトップアプリです。たとえば、メール返信文、問い合わせ対応のテンプレート、よく使う説明文などを `.txt` ファイルとして管理できます。

初学者向けに言うと、「ファイルを読み書きする Python プログラム」に GUI を付けた作品です。画面左側に定型文ファイルの一覧を表示し、右側で内容を編集します。保存ボタンでテキストファイルに書き込み、コピーボタンでクリップボードに送ります。

📄 **コード**: [teikei_kanri.py](./teikei_kanri.py)

### 利用者の流れ

1. アプリを起動すると、左側に保存済みテンプレートの一覧が表示されます。
2. テンプレートをクリックすると、右側の編集欄に内容が表示されます。
3. 文章を編集して「保存」を押すと、テキストファイルに上書き保存されます。
4. 「コピー」を押すと、本文がクリップボードに入り、メールなどへ貼り付けできます。
5. 「新規作成」で新しいテンプレートファイルを追加できます。

### 裏側の処理

アプリ起動時に `template-files/` フォルダを確認し、なければ自動で作成します。そのフォルダ内の `.txt` ファイルを一覧として読み込み、ファイル名順に表示します。選択されたファイルの内容は `Path.read_text()` で読み込み、保存時は `Path.write_text()` で書き込みます。

文字コードの違いでファイルが読めない問題を避けるため、`utf-8`、`utf-8-sig`、`cp932` の順に読み込みを試しています。Windows の古いメモ帳や Excel 由来のテキストにも対応しやすくしています。

### コードを読む順番

| 順番 | 関数・メソッド | 初学者向けの見どころ |
|------|----------------|----------------------|
| 1 | `main()` | Flet が用意した `page` にウィンドウサイズを設定し、アプリ本体を起動する入口 |
| 2 | `TemplateManager.__init__()` | 保存フォルダ、現在編集中ファイル、画面部品を準備する初期化処理 |
| 3 | `setup_ui()` | 左右2ペインの画面を作り、ボタンと処理を `on_click` でつなぐ部分 |
| 4 | `load_templates()` / `select_template()` | ファイル一覧を読み込み、選択されたファイル内容をテキスト欄に表示する流れ |
| 5 | `read_file()` / `save_template()` | 文字コードを試しながら読み込み、UTF-8 で保存するファイル I/O |
| 6 | `create_template()` / `delete_template()` | ダイアログを使った新規作成・削除確認の実装 |
| 7 | `copy_content()` / `show_snackbar()` | クリップボード連携と、操作結果を画面に通知する共通処理 |

### 処理の流れをコードで追う例

テンプレートを開く処理は、左側リストの `ListTile` から始まります。クリックすると `select_template()` が呼ばれ、`template-files/` の中にある対象ファイルを `read_file()` で読み込みます。読み込んだ文字列は `self.text_field.value` に入るため、右側の編集欄に表示されます。

保存処理は `save_template()` で行います。`self.current_file` に「今開いているファイル」が入っているかを確認し、入っていれば `write_text()` で保存します。どのボタンを押しても最後に `show_snackbar()` を使って結果を知らせるため、成功・警告・エラーの表示方法が1か所にまとまっています。

### アーキテクチャ

- **単一クラス構成**: `TemplateManager` クラスに画面組み立て・ファイル I/O・イベント処理を集約
- **保存形式**: スクリプトと同階層の `template-files/` に `.txt` ファイルとして保存
- **状態管理**: 編集中ファイルのパスを `self.current_file` に保持
- **GUI**: Flet を使い、左側の一覧と右側の編集エリアを作成

### 主な実装機能

| 機能 | 初学者向けの説明 |
|------|----------------|
| 一覧表示 | フォルダ内のファイルを探し、画面左側にリスト表示 |
| 内容編集 | 複数行入力できるテキスト欄で文章を編集 |
| 新規作成 | ダイアログでファイル名を入力し、新しい `.txt` を作成 |
| 保存 | 編集内容を UTF-8 のテキストファイルとして保存 |
| 削除 | 確認ダイアログ後にファイルを削除 |
| コピー | `pyperclip.copy()` で本文をクリップボードへ送信 |
| 通知 | 保存成功・警告・エラーを色付きメッセージで表示 |

### この作品で学べること

- Python でフォルダやファイルを扱う方法
- GUI アプリの画面部品とイベント処理の関係
- 文字コードの違いに対応する考え方
- クリップボード連携など、OS 機能との連携
- 小さな業務改善ツールを自分で作る流れ

```bash
pip install flet pyperclip
python teikei_kanri.py
```

## ④ 付箋アプリ（Python / tkinter）

付箋アプリは、デスクトップ上に複数の付箋メモを表示し、内容・色・位置を保存できる GUI アプリです。タスクメモ、買い物リスト、作業中の注意点などを画面上に置いておく用途を想定しています。

初学者向けに言うと、「メモ帳のような小さなウィンドウを複数作り、それぞれの状態を JSON に保存する」作品です。Python 標準ライブラリの `tkinter` だけで画面を作っているため、追加ライブラリなしで動作します。

📄 **コード**: [sticky_notes.py](./sticky_notes.py)

### 利用者の流れ

1. アプリを起動すると、付箋を管理するメイン画面が表示されます。
2. 「新規作成」を押すと、新しい付箋ウィンドウが開きます。
3. タイトルや本文を入力すると、自動的に保存されます。
4. 色変更で付箋ごとに背景色を変えられます。
5. アプリを終了しても、次回起動時に保存済み付箋を復元できます。

### 裏側の処理

付箋ごとの情報は、ID・タイトル・本文・色・座標・更新時刻として管理しています。保存時には、それらを `sticky_notes_data.json` にまとめて書き出します。JSON は人間にも読みやすいテキスト形式なので、データの構造を確認しやすい点が学習向きです。

キー入力のたびに保存処理を呼び出すことで、「保存ボタンを押し忘れて内容が消える」リスクを下げています。また、付箋ウィンドウが開いているか閉じているかを判定し、一覧画面に状態として表示します。

### アーキテクチャ

| クラス | 役割 |
|------|------|
| `StickyNote` | 個別の付箋ウィンドウと、その付箋のデータ・操作を担当 |
| `StickyNotesApp` | メイン画面、付箋一覧、全付箋の保存・復元を担当 |

この2クラス構成にすることで、「1枚の付箋に関する処理」と「アプリ全体を管理する処理」を分けています。

### コードを読む順番

| 順番 | クラス・メソッド | 初学者向けの見どころ |
|------|------------------|----------------------|
| 1 | `main()` | `tk.Tk()` でメインウィンドウを作り、`mainloop()` でイベント待ち受けを始める入口 |
| 2 | `StickyNotesApp.__init__()` | 全付箋を入れる辞書、次に使う ID、保存ファイル名を準備する |

| 5 | `StickyNote.create_window()` | 個別の付箋ウィンドウ、タイトル入力欄、本文入力欄、色変更ボタンを作る |

| 8 | `update_note_list()` / `update_stats()` | 内部データを画面の一覧・統計表示へ反映する処理 |

### 処理の流れをコードで追う例

新しい付箋を作る処理は、メイン画面の「新規作成」ボタンから `StickyNotesApp.add_note()` に進みます。ここで `StickyNote` クラスのインスタンスを作り、`note.create_window()` で実際の付箋ウィンドウを表示します。作った付箋は `self.notes` という辞書に `ID: 付箋オブジェクト` の形で保存されます。

自動保存は、入力欄にキー入力があるたびに `on_text_change()` や `on_title_change()` が呼ばれ、親アプリの `auto_save()` へつながります。`save_notes()` では各付箋からタイトル・本文・色・位置を取り出して JSON に変換します。`load_notes()` はその逆で、JSON から付箋オブジェクトを作り直します。

## ⑥ サーバー監視ダッシュボード（Python / Flask / psutil / Chart.js）

サーバー監視ダッシュボードは、PC やサーバーの状態をブラウザで確認できる運用支援ツールです。CPU 使用率、メモリ使用量、ディスク使用率、ネットワーク通信量、動作中プロセスなどをリアルタイムに表示します。

初学者向けに言うと、「Python でパソコンの状態を取得し、それを Web ページに表示する」作品です。Python の Flask が Web サーバーとして動き、`psutil` が OS から情報を取得し、JavaScript と Chart.js がブラウザ上でグラフや数値を更新します。

![サーバー監視ダッシュボード](https://raw.githubusercontent.com/ns7jp/server-monitor/main/screenshot.png)

📄 **コード**: [ns7jp/server-monitor](https://github.com/ns7jp/server-monitor)

### 利用者の流れ

1. `python app.py` で Flask アプリを起動します。
2. ブラウザで `http://localhost:5000/` を開きます。
3. ダッシュボードに CPU・メモリ・ディスクなどの状態が表示されます。
4. 2秒ごとに統計情報が更新され、直近60秒の変化がグラフで見られます。
5. CPU 使用率が高いプロセスを一覧で確認できます。

### 裏側の処理

Flask は `/` でダッシュボード画面を返し、`/api/stats` と `/api/processes` で JSON データを返します。ブラウザ側の JavaScript は Fetch API を使って定期的に JSON を取りに行き、受け取った値を画面に反映します。

`psutil` は OS から CPU 使用率、メモリ容量、ディスク使用率、ネットワーク送受信量、プロセス一覧を取得します。Chart.js は、その数値を折れ線グラフとして描画します。WebSocket ではなく定期ポーリング方式にすることで、初学者にも追いやすい実装にしています。

### 主な実装機能

| 機能 | 初学者向けの説明 |
|------|----------------|
| システム情報 | OS・ホスト名・起動時刻・稼働時間を表示 |
| CPU 使用率 | 現在どれくらい CPU が使われているかを数値とグラフで表示 |
| メモリ・スワップ | メモリ容量、使用量、空き容量を表示 |
| ディスク | 各パーティションの使用率を表示し、使用率に応じて色分け |
| ネットワーク I/O | 送受信量や通信速度を表示 |
| 履歴グラフ | CPU / メモリの直近60秒の推移を Chart.js で表示 |
| プロセス一覧 | CPU を多く使っているプロセス TOP15 を表示 |
| 自動更新 | 統計は2秒ごと、プロセスは5秒ごとに非同期更新 |

### アーキテクチャ

```text
Browser
  ├─ Chart.js で履歴グラフを描画
  └─ Fetch API で定期更新
        ↓
Flask app
  ├─ /api/stats
  └─ /api/processes
        ↓
psutil
  └─ OS からCPU・メモリ・ディスク・ネットワーク・プロセス情報を取得
```

### 技術的な工夫

- **しきい値ベースの色分け**: 使用率が高くなるほど警告色に近づき、異常に気づきやすい表示にした
- **CPU取得の安定化**: `psutil.cpu_percent(interval=0.5)` で初回取得の 0% 固定を回避
- **ローリング履歴**: 過去30点（約60秒分）だけを保持し、グラフが長くなりすぎないように整理
- **安全側の初期値**: 学習用の初期設定は `host='127.0.0.1'`、`debug=False` とし、自分の PC からだけ確認できる設定にした
- **構成の分離**: Flask 側はデータ取得、JavaScript 側は表示更新という役割分担にした

### コードを読む順番

| 順番 | ファイル | 初学者向けの見どころ |
|------|----------|----------------------|
| 1 | `app.py` | Flask のルーティング、`psutil` での情報取得、JSON API の返し方 |
| 2 | `templates/index.html` | ダッシュボードの画面構造。カード、表、グラフ用 `canvas` の配置 |
| 3 | `static/js/dashboard.js` | Fetch API、定期更新、Chart.js の初期化とデータ差し替え |
| 4 | `static/css/style.css` | メトリクスカード、グラフ、警告色、レスポンシブ表示の見た目 |
| 5 | `requirements.txt` | Flask / psutil など、実行に必要な外部ライブラリ一覧 |

### 処理の流れをコードで追う例

CPU 使用率の表示は、`static/js/dashboard.js` の `fetchStats()` から追うと理解しやすいです。この関数が `/api/stats` にアクセスし、Flask 側の `app.py` にある `stats()` が呼ばれます。`stats()` は `psutil.cpu_percent()` や `psutil.virtual_memory()` で値を集め、`jsonify()` でブラウザに返します。

ブラウザは受け取った JSON を使って、数値カードを書き換え、`updateChart()` で Chart.js のグラフデータを更新します。ここでは「Python はデータ取得」「Flask は API として返す」「JavaScript は画面更新」という役割に分かれています。

### この作品で学べること

- Flask で画面と JSON API を作る方法
- Python で OS の情報を取得する方法
- Fetch API で定期的にデータを取得する方法
- Chart.js で時系列グラフを描画する方法
- サーバー監視・インフラ運用の入口となる考え方
